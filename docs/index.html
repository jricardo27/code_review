Tips before asking for a Code Review...

Before asking for a Code Review please do one yourself over your own Pull Request (PR).

The following is a list of the most frequent flaws found in code reviews.

These rules align with PEP8 and in some cases discard some of the alternatives proposed as the others are better.
For example the following two cases are considered valid by PEP8: 

foo = long_function_name(
    var_one, var_two, var_three, var_four)

foo = long_function_name(var_one, var_two,
                         var_three, var_four)

But for our purposes the second example is not valid as:

  1. It would require re-indenting if the variable or the function name change.
  2. It wastes space by over-indenting.

Copy Pasta

Copying code is one of the easies ways to achieve results, however at the expense of maintainability.
The moment you find yourself copying code, stop for a moment and consider if you can do better and factor out the code.

The following is a list of common scenarios when you duplicate code and the most common ways to deal with it:

  • Extract it to a base class

    So, you find yourself copying a Django View class (could be any other class) which already implement some useful things (e.g. authentication, permissions, etc) why not extract those useful methods into a Base Class or a Class Mixin?
    It's easy: 

    # BAD
class AmazingView(TemplateView):
    """Display amazing stuff."""
    template = 'amazing.html'

    def amazing_method(self):
        """Do something amazing."""
        pass

    def dispatch(self):
        """Do something amazing with the output."""

        self.amazing_method()


class NotSoAmazingView(TemplateView):
    """Display amazing stuff."""

    template = 'not_so_amazing.html'

    def amazing_method(self):
        """Do something amazing."""
        pass

    def dispatch(self):
        """Do something amazing with the output."""

        self.amazing_method()

    # GOOD
class AmazingBaseClass(TemplateView):
    """Base class for displaying amazing stuff."""

    def amazing_method(self):
        """Do something amazing."""
        pass

    def dispatch(self):
        """Do something amazing with the output."""

        self.amazing_method()


class AmazingView(AmazingBaseClass):
    """Display amazing stuff."""

    template = 'amazing.html'


class NowImAlsoAnAmazingView(AmazingBaseClass):
    """Display some more amazing stuff."""

    template = 'some_more_amazingness.html'
  • Factor out common code

    This time you need to perform certain operation, you found out that there is already a method/function that do almost what you need, however it does in green not in blue, so why not just factor out the code into a tird method/function and pass the color a an argument?
    It's easy: 

    # BAD
def color_text_in_green(text):
    """Color text in green."""

    return '<{color}>{text}<{color}>'.format(
        color='green',
        text=text,
    )

def color_text_in_blue(text):
    """Color text in blue."""

    return '<{color}>{text}<{color}>'.format(
        color='blue',
        text=text,
    )

print(color_text_in_green('Grass'))
print(color_text_in_blue('Sea'))

    # GOOD
def color_text(text, color):
    """Apply the given color to the text."""

    return '<{color}>{text}<{color}>'.format(
        color=color,
        text=text,
    )

def color_text_in_green(text):
    """Color text in green."""

    return color_text(text, 'green')

def color_text_in_blue(text):
    """Color text in blue."""

    return color_text(text, 'blue')

print(color_text_in_green('Grass'))
print(color_text_in_blue('Sea'))
  • Template includes

    Now you have a bunch of templates to do, easy peasy, they are all the same, just the title is different. The above case sounds like the perfect scenario to use the Django template tag include: 

    <!-- BAD -->

<!-- model1_details.html -->
{% extends "app_base.html" %}

{% block breadcrumbs %}
    Model1
{% endblock %}

{% block content %}
    <h1>Model1 -- {{ object.name }}</h1>
    <table>
        <theader>
            <tr>
                {% for header object.headers %}
                    <th>header</th>
                {% endfor %}
            </tr>
        </theader>
        <tbody>
            <tr>
                {% for value object.values %}
                    <td>value</td>
                {% endfor %}
            </tr>
        </tbody>
    </table>
{% endblock %}


<!-- model2_details.html -->
{% extends "app_base.html" %}

{% block breadcrumbs %}
    Model2
{% endblock %}

{% block content %}
    <h1>Model2 -- {{ object.name }}</h1>
    <table>
        <theader>
            <tr>
                {% for header object.headers %}
                    <th>header</th>
                {% endfor %}
            </tr>
        </theader>
        <tbody>
            <tr>
                {% for value object.values %}
                    <td>value</td>
                {% endfor %}
            </tr>
        </tbody>
    </table>
{% endblock %}

    <!-- GOOD -->

<!-- includes/model_details.html -->
<h1>{{ model_name }} -- {{ object.name }}</h1>
<table>
    <theader>
        <tr>
            {% for header object.headers %}
                <th>header</th>
            {% endfor %}
        </tr>
    </theader>
    <tbody>
        <tr>
            {% for value object.values %}
                <td>value</td>
            {% endfor %}
        </tr>
    </tbody>
</table>


<!-- model1_details.html -->
{% extends "app_base.html" %}

{% block breadcrumbs %}
    Model1
{% endblock %}

{% block content %}
    {% include "includes/model_details.html" with model_name='Model1' %}
{% endblock %}


<!-- model2_details.html -->
{% extends "app_base.html" %}

{% block breadcrumbs %}
    Model2
{% endblock %}

{% block content %}
    {% include "includes/model_details.html" with model_name='Model2' %}
{% endblock %}

Code lay-out

Formatting problems are easy to overlook but they are the most common.
Some of them are highlighted by tools like pylint but is not always the case.
These small omissions cause a lot of going back and forth wasting the reviewers precious time.

Python

Trailing commas

Leaving a trailing comma on the last entry is useful because then every line has the same format and adding entries to the end is the same as adding them to the middle.

Dictionaries 

# GOOD.
my_dict = {
    'A': 1,
    'B': 2,
}


# BAD.
my_dict = {
    'A': 1,
    'B': 2
}

# BAD.
# This is bad with or without the trailing comma as it's hard to read.
my_dict = {
    'A': 1, 'B': 2,
}
Calling Functions 
# GOOD.
call_some_function(
    param1,
    param2,
)

# GOOD.
# If the function doesn't expect more parameters, a closing parenthesis `)`
# can be placed at the end of the last parameter.
call_some_function(
    param1,
    param2)

# GOOD.
call_some_function(
    param1, param2)


# BAD.
call_some_function(
    param1, param2
)

# BAD.
call_some_function(
    param1,
    param2
)

# BAD.
call_some_function(param1,
    param2,
)

# BAD.
call_some_function(param1,
    param2)

Breaking long lines

PEP8 recommend limiting all lines to a maximum of 79 characters.
However there are some projects that raised the limit to 119 characters.
My recommendation is to stick to 79 characters even when working on projects with a longer limit and only use the extra limit when breaking the line makes it harder to read.

Even though PEP8 considers backslashes appropriate in some cases. They are totally discouraged in this guide.
The reason for that is that other alternatives are better and errors could be caused if a space is added after the backslash.
Alternatives to break long lines: 

# Parenthesis. Case 1.
if (something == True and
        something_else == False):


# Parenthesis. Case 2.
string = (
    'This string is too long to fit.'
)

string = (
    'This string is too long to fit.'
    ' This one is even longer.'
)


# Parenthesis. Case 3.
assert val > 4, (
    "val is too small")

assert val > 4, (
    "val is too small"
)

Gherkin (Aloe tests)

The main purpose of Gherkin is to allow writing tests from the point of view of the user using an easy to understand language.

Before asking for a code review, read the steps aloud to yourself, they should sound grammatically coherent. 

Feature: Examples of grammar in Scenarios.

    As a developer
    I should write coherent and grammatically correct scenarios
    So that the attention focuses on what the test does and not how bad it sounds.

    Scenario: A good scenario.
        Given there is a user "username" in the database
        When I login with "username"
        Then the page title should be "Welcome"

    Scenario: A bad scenario.
        And there is a user "username" in the database
        Given I login with "username"
        Then the page title should be "Welcome"

    Scenario: Another bad scenario.
        Given there is a user "username" in the database
        Given I login with "username"
        Then the page title should be "Welcome"


Recommendations:
  1. Try to perform all the application setup before logging in. e,g:
    • Create model objects.
    • Load fixtures.
  2. Prefer using `Given` for setup steps. It is "ok" to use it for user actions defined in the `Background`.
  3. Separate sets of actions with a blank line: 
    When I login with "username"
Then the page title should be "Welcome"

When I click "My Account"
Then I should see "My name"

When I press "Edit"
And I fill in "Name" with "Another name"
And I press "Save"
Then I should see "Changes saved succesfully"
And I should see "Another name"

HTML

HTML is a messy language, our efforts should go towards make it if not less messy at least easier to read.

Recommendations
  1. Indentation makes it easier to read.
  2. Align attributes and use extra indentation if necessary.
  3. Try to keep lines short, lines can be break by:
    • Placing closing tag on the next line.
    • Putting attributes in multiple lines.
    • Placing the content on the next line.
  4. Do not use inline styles. Use CSS classes instead.
  5. When possible, create small templates (Django) and include then to avoid repeating code. Remember that you can set variables. See include.
Bad Example:
1  {% for object in objects %}
2  <div class="box" aria-label="Object details" more-attributes="true" even-more-attributes="false">
3      <h2>{{ obj.title }}</h2>
4      <div id="contentId"
5           class="content">
6          {{ obj.description}}
7          <div style="display: none">
8              {{ obj.extra_details }}
9          </div>
10     </div>
11 </div>
12 {% endfor %}


Lines 1-2, No indentation after `{{ for }}`.
Line 2, Attributes in one line are harder to
    read.
Line 2, If adding more attributes the line
    gets even longer.
Lines 4-6, No extra indentation of attributes
    makes hard to notice where the div content
    is.
Line 6, Variable tag is not formatted
    correctly: a space is missing before the
    closing `}}`
Line 7, Inline styles are not recommended as
    they are hard to maintain. CSS class
    should be use instead.
Good Example:
1  {% for object in objects %}
2      <div    class="box"
3              aria-label="Object details"
4              more-attributes="true"
5              even-more-attributes="false"
6      >
7          <h2>{{ obj.title }}</h2>
8          <div    id="contentId"
9                  class="content">
10             {{ obj.description }}
11             <div class="hidden">
12                 {{ obj.extra_details }}
13             </div>
14         </div>
15     </div>
16 {% endfor %}


Line 1-2, All code is indented after
    `{{ for }}` making it easier to see what's
    inside and where the block ends.
Lines 2-5, Attributes are aligned and have
    one more extra indentation.
Line 6, Even though uncommon, closing the
    tag on a separate line makes it easier to
    add more attributes.
    It also adds a blank space between the
    tag and it's content.
    It's also OK to close the tag at the end
    of the last attribute (Line 9).
Line 11, A CSS class is used instead of an
    inline style.
